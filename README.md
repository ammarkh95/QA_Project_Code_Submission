# Embedded Thermal Control System - Firmware Test Code Sample

## Introduction

This repository contains the quality strategy and automated test cases code for the thermal control system.

The system consists of a PWM fan with a tachometer feedback pin, I2C based temperature sensor and MCU. The MCU uses PWM signal to control the speed of the fan. The tachometer pin signal allows the MCU to measure the actual RPM speed of the fan to make sure it is closely tracking the set point RPM speed for the target temperature.

## Firmware Simulation in Python

In absence of real hardware, the firmware behavior is mocked with a Python class that interface with the test framework to demonstrate the logic and execution workflow of automated test cases.

The Python module `thermal_control_sim_server.py` implements a server process based on Python [multiprocessing](https://docs.python.org/3/library/multiprocessing.html#module-multiprocessing.connection) module. The server process will act as the firmware recieving commands over UART from the test setup (i.e. client) and replying back to it. During the test run the test framework will connect to the exposed server process and interact with it to verify the expected logic.

The server simulation process will accept the following messages strucutre:
  - "GET" <"Parameter Name"> (e.g. TEMPERATURE; TACHOMETER_RPM, PWM_OUTPUT_DUTY)
  - "SET" <"Parameter Name"> (e.g. TARGET_TEMPERATURE, PWM_OUTPUT_DUTY)
  - "TRIGGER_FAULT" <"FAULT ERROR CODE"> (e.g. FAULT FAN FEEDBACK ERROR)
  - "EXECUTE_CONTROL_LOOP" <"NUMBER OF ITERATIONS"> (e.g. 5)

*Please note that this is a great simplification of a real time embedded system and in reality hardware in the loop testing will be needed to validate the acutal behavior of the system.*

## Running the Test Cases
For running the test cases, the following pre-requisites are nedded and recommended:

- Python 3.12 installed on Windows / Linux / MacOS platform
- Create and activate Python virtual environemnt to isolate testing dependencies:

  ```bash
  python3 -m venv venv # (or) python -m venv venv (on Windows)
  . venv/bin/activate  # (or) . venv/Sripts/Activate.ps1 (on Windows)
  ```
- Install required test framework dependencies (e.g. pytest, Robot)

  ```bash
  pip install -r requirements.txt
  ```

- Execute validation test suite with Robot frameowrk

  ```bash
  # Robot Framework front-end
  robot --outputdir test_results thermal_control_validation.robot
  ```

## Logging and Tolerance Check via Cli Script
Examples below to run the command line interface script
```bash
python simulator_cli.py sweep --step 10 --out logs/pwm_vs_rpm_sweep.csv
python simulator_cli.py check --duty 60 --tolerance 10 
```

## CI/CD Firmware Validation Pipeline
To automate the execution of test cases and publishing of test results, a CI/CD workflow is implemented in `azure-pipelines-validation.yml`
The pipeline is written for Azure Pipelines Yaml structure and is intended to run on a self-hosted agent such that automated hardware-in-the-loop testing is supported.

The pipelines executes the following tasks:

  - Checking out the git repository code
  - Selecting the Python interpreter version (several versions might be installed and used on the hosted test agent)
  - Creating and activating a Python virtual environment
  - Installing Python test dependencies
  - Executing Robot Validation test suite (adjustments can be made to run a subset of tests)
  - Upload test results and report generated by Robot framework as test artifacts 

The following customizations and improvements could be made to the pipeline:
  - Setting the pipeline to trigger test execution when firmware build pipeline succeeds (i.e new firmware build available for testing)
  - Link the test results and report inside the test managment tool for better traceability and post-processing of testing metrics
  - Parametrize input to the pipeline for better control of test execution (e.g run specific test suite for pull requests)
